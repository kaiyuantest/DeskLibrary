# 项目背景与重构原因

本文档用于给新的 AI 或开发者快速解释三件事：

1. 当前仓库里为什么同时存在旧项目和新项目文档
2. 旧项目到底是什么
3. 新项目为什么不能只在旧结构上继续补功能

## 2. 旧项目是什么

旧项目本质上是一个“桌面收藏工作台”，主要能力包括：

- 剪贴板收藏
- 图片收藏
- 文件 / 文件夹资源库
- 浏览器 Cookie 卡片
- 截图 OCR / 翻译
- 临时便签
- 悬浮图标 / 悬浮菜单 / 托盘常驻

如果从用户角度理解，它是一个：

“把桌面上临时出现的碎片信息收集起来，并提供二次使用能力的 Windows 工具箱。”

## 3. 旧项目当前的数据能力

旧项目已经涉及的数据，不只是文本、链接、图片，还包括：

- 收藏记录
- 图片文件
- OCR 文本
- 链接
- 文件 / 文件夹资源
- 浏览器卡片
- Cookie 快照
- 临时便签
- 应用设置

换句话说，旧项目已经天然接近“个人桌面信息资产管理器”。

## 4. 为什么要重构

重构不是因为旧项目功能太少，而是因为产品目标已经变了。

新的目标不是继续做一个“桌面收藏工具”，而是做一个：

“本地个人数据仓 + AI 检索与整理代理”

两者的差别非常大。

### 4.1 旧目标

旧项目更偏：

- 采集
- 收藏
- 快速查看
- 手工管理

### 4.2 新目标

新项目更偏：

- 统一沉淀所有个人数据到本地文件夹
- 让 AI 能稳定理解这些数据
- 用户用自然语言检索、整理、归纳、改写、归档
- AI 还能在受控规则下做增删改查和备份

所以，新项目的核心不是“多几个功能按钮”，而是“重新定义主系统”。

## 5. 为什么不能只在旧结构上继续堆功能

旧项目当前存储模型大致是：

- Electron `userData` 目录
- 多个 JSON 文件
- 各功能模块分别读写自己的数据文件

这个结构能支持桌面工具，但不适合直接演进成 AI 原生数据仓，主要问题有：

### 5.1 文件夹不是主系统

旧项目的数据目录更像软件内部实现细节，而不是用户可长期控制的主数据仓。

### 5.2 没有统一文档层

文本、图片、文件、Cookie 卡片各有各的数据结构，AI 很难以统一方式检索和理解。

### 5.3 没有 `raw / staging / final` 三层

如果直接让 AI 修改旧项目现有 JSON，很容易污染原始数据。

### 5.4 没有 AI 受控执行协议

旧项目没有设计“AI 能做什么、不能做什么、什么动作要确认”的规则层。

### 5.5 没有长期标准格式

旧项目的存储格式是为了当前软件跑通，不是为了未来其他工具读取。

## 6. 已经明确的新方向

新方向已经通过前面对话确认，核心是：

1. 本地数据文件夹是主入口
2. 桌面软件只是采集器、管理器、执行器
3. AI 既支持内置聊天，也支持外部 AI 面向文件夹工作
4. AI 重点是整理和归纳，不只是搜索
5. AI 可以增删改查，但必须走受控协议
6. 原始数据、AI 工作数据、正式数据必须分层
7. 高风险动作必须确认
8. 文件夹格式要长期稳定，未来别的工具也能读

## 7. 新项目第一阶段的数据范围

第一阶段已经确认支持：

- 文本
- 图片
- 链接
- 文件
- 浏览器 Cookie / 卡片

其中 Cookie 在第一阶段的定位是：

- 站点登录卡片的一部分
- 归档对象
- 辅助执行对象

而不是第一版的核心语义理解对象。

## 8. 当前这套文档的用途

`DeskLibrary.Next` 目录的文档不是论文，也不是 PRD 展示稿，而是：

- 给新开的 AI 对话提供上下文
- 给后续编码提供边界约束
- 给架构演进提供稳定协议

因此这里的文档重点不是营销，而是：

- 说清楚“是什么”
- 说清楚“为什么”
- 说清楚“怎么落”

## 9. 后续实现时的基本原则

以后如果要继续写代码，应遵守这些原则：

1. 不要再把新能力直接耦合进旧项目原有 JSON 结构
2. 先实现新数据仓协议，再接旧能力迁移
3. 先实现目录协议和操作协议，再做复杂界面
4. AI 相关能力必须先有规则层和操作层
5. 任何正式数据都要能回溯到原始数据

## 10. 当前结论

如果你是新的 AI 或新的开发者，请先记住一句话：

“旧项目是能力来源，新项目是架构重建；不要把 `DeskLibrary.Next` 当成旧 README 的附录，而要把它当成下一代产品的主设计入口。”
