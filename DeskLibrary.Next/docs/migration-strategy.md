# 旧项目迁移策略

本文档用于回答一个很实际的问题：

“既然旧项目已经有这么多能力，新项目到底该怎么迁？”

这里的重点不是一次性重写全部代码，而是定义：

- 哪些能力可以复用
- 哪些结构必须放弃
- 迁移顺序怎么排

## 1. 迁移基本原则

新项目的迁移必须遵守下面几条：

1. 先立协议，再迁代码
2. 先立数据仓，再接采集器
3. 先立执行控制，再接 AI
4. 旧能力可以复用，旧存储结构不能直接延续

换句话说：

- 可以复用旧项目的业务理解
- 可以复用旧项目的部分采集代码
- 可以复用旧项目的浏览器卡片处理能力
- 但不能继续把旧项目的 JSON 文件结构当新项目主模型

## 2. 旧项目里最值得复用的部分

### 2.1 文本 / 图片采集语义

旧项目已经具备：

- 剪贴板文本捕获
- 剪贴板图片捕获
- 来源应用 / 窗口标题语义
- 收藏方式 / 分类语义

这些业务语义很有价值，可以迁到新对象模型中。

### 2.2 文件导入与备份语义

旧项目已经定义了：

- `link`
- `backup`

这两个模式在新项目里仍然成立，应该继续保留。

### 2.3 浏览器卡片能力

旧项目在浏览器卡片这块已经积累了较多实际逻辑，包括：

- Chrome 用户目录导入
- 自建浏览器目录导入
- BitBrowser API 导入
- Cookie 卡片刷新 / 注入

这些可以继续作为能力来源，但对象输出必须改造成新协议。

### 2.4 Electron 桌面壳

旧项目已经有：

- 主窗口
- 托盘
- 悬浮图标
- 悬浮菜单
- 截图相关页面

这些不是当前第一阶段的核心阻塞项，但可以作为未来桌面端承载层继续复用。

## 3. 旧项目里不能直接沿用的部分

### 3.1 `userData + 多 JSON 文件` 主存储模型

旧项目当前主存储偏向内部实现，不适合作为长期标准格式。

### 3.2 按功能模块分散定义对象

新项目必须把对象统一收敛到：

- 原始对象
- 标准化文档
- 操作对象
- 快照对象

### 3.3 缺少执行控制的直接写数据模式

旧项目是普通本地工具写法，新项目必须引入：

- 风险分级
- 操作队列
- 确认流
- 快照
- 历史

### 3.4 缺少统一文档层

旧项目没有 `Document` 这一层，新项目必须建立。

## 4. 推荐迁移思路

### 阶段 1：协议先行

先实现：

- 新数据仓目录结构
- 新对象模型
- 新操作协议

此阶段目标不是做完 UI，而是先让系统有稳定骨架。

### 阶段 2：接最小采集链路

先把最值钱的旧能力接进新协议：

- 文本采集
- 图片采集
- 文件导入

让这些数据能直接进入：

- `raw`
- `staging`
- `final`

### 阶段 3：接索引与检索

建立：

- SQLite 元数据索引
- 全文检索
- 文本 / 图片向量索引

### 阶段 4：接 AI 整理链路

建立：

- `staging/document`
- 标签建议
- 摘要建议
- 专题建议
- `publish_to_final`

### 阶段 5：接浏览器卡片迁移

把旧项目浏览器卡片逻辑改造成：

- 新 `BrowserCard` 对象
- 新操作协议
- 新快照机制

### 阶段 6：接桌面 UI

最后再逐步重接：

- 主界面
- 内置聊天
- 高风险确认界面
- 历史和回滚界面

## 5. 为什么不建议一开始就迁全部功能

因为如果一开始就把：

- 悬浮菜单
- 截图
- OCR
- 浏览器卡片
- 便签
- 托盘

全部绑到新架构上，很容易出现两种问题：

1. 架构还没稳定，功能先把结构拖乱
2. 新项目还没立住，旧项目细节已经绑死设计

所以第一阶段最合理的策略是：

- 先把高价值核心链路跑通
- 再把复杂辅助功能接回去

## 6. 旧数据兼容的建议思路

以后如果要兼容旧数据，建议不要让新系统直接以旧 JSON 为运行态主结构。

更稳的方式是：

1. 读取旧 JSON
2. 转换成新对象模型
3. 写入新数据仓
4. 建立迁移日志

也就是说，用“导入迁移”而不是“继续原地读旧结构”。

## 8. 当前最推荐的迁移起点

如果下一步要从文档进入实现，最推荐的起点是：

1. 在 `DeskLibrary.Next` 下建立新代码结构
2. 先实现 `vault-core`
3. 先做文本 / 图片 / 文件三类对象落盘
4. 再做 `staging` 和 `final`
5. 再做操作执行器

浏览器卡片可以放在第一阶段后半段接入，不是第一个阻塞项。

## 9. 当前结论

迁移的正确姿势不是“把旧项目改一改”，而是：

“先建立新主系统，再把旧项目的有效能力逐步接进来。”
