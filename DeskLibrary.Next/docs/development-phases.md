# 开发阶段与优先级

本文档用于说明：

- 新项目应该分几步做
- 每一步的目标是什么
- 什么是当前真正的阻塞项

## 1. 总原则

新项目不能按“想到一个功能做一个功能”的方式推进。

更合适的做法是按阶段推进，每个阶段都围绕一个明确目标：

- 先把骨架立住
- 再把核心链路跑通
- 再把复杂能力接回去

## 2. 阶段 0：文档定型

当前所在阶段基本属于这一阶段。

目标：

- 把新项目的方向、边界、协议写清楚
- 避免后续边实现边推翻

当前已完成的主要内容：

- 项目背景
- 产品定义
- 数据仓目录结构
- 数据对象模型
- 总体架构
- AI 操作协议

## 3. 阶段 1：数据仓骨架

这是最优先阶段。

目标：

- 真正建立新数据仓的最小可运行骨架

此阶段必须实现：

1. `vault.json`
2. `raw` 目录初始化
3. `staging` 目录初始化
4. `final` 目录初始化
5. `operations` 目录初始化
6. 基础对象读写 API
7. 基础 ID 规则
8. 基础版本字段

验收标准：

- 可以创建一个合法数据仓
- 可以写入文本、图片、文件基础对象
- 可以读取并列出这些对象

## 4. 阶段 2：核心对象落盘

目标：

- 先把第一阶段必须支持的对象真正写进新协议

优先对象：

1. `Record`
2. `ImageAsset`
3. `FileAsset`
4. `LinkRecord`
5. `BrowserCard`

验收标准：

- 这五类对象都可以合法落盘
- 都能回读
- 都有基础索引入口

## 5. 阶段 3：AI 工作层

目标：

- 让 AI 至少能在 `staging` 工作，而不是只会读 `raw`

此阶段必须实现：

1. `Document`
2. `Task`
3. 基础标签建议
4. 基础摘要建议
5. `staging -> final` 发布接口

验收标准：

- 原始文本和图片可以生成 `staging/document`
- 用户确认后可以生成 `final/document`

## 6. 阶段 4：操作执行引擎

目标：

- 把 AI 的执行行为管起来

此阶段必须实现：

1. `Operation`
2. 风险分级
3. 确认流
4. `Snapshot`
5. 历史记录

验收标准：

- 低风险动作可自动执行
- 高风险动作必须挂起等待确认
- 执行前能生成快照

## 7. 阶段 5：索引与检索

目标：

- 让系统具备真正可用的检索能力

此阶段必须实现：

1. SQLite 元数据索引
2. 全文检索
3. 文本语义检索
4. 图片视觉语义检索入口
5. 结果融合策略

验收标准：

- 可以找文本
- 可以找图片
- 可以按主题召回相关内容

## 8. 阶段 6：旧能力迁入

目标：

- 把旧项目里已验证过的能力接到新协议上

优先顺序建议：

1. 剪贴板文本
2. 剪贴板图片
3. 文件导入
4. 浏览器卡片
5. OCR / 截图

不建议第一时间迁：

- 悬浮菜单
- 临时便签
- 复杂托盘交互

## 9. 阶段 7：桌面 UI 与 AI 界面

目标：

- 让用户能真正使用新系统

此阶段可以实现：

- 数据仓选择界面
- 内置聊天界面
- 命中结果列表
- 高风险确认弹窗
- 历史和回滚界面

## 10. 阶段 8：高级整理能力

目标：

- 把“整理助手”的价值真正拉出来

包括：

- 专题生成
- 自动归档建议
- 集合与主题管理
- 批量整理计划
- 更精细的图片理解能力

## 11. 当前真正的阻塞项

如果今天开始写代码，真正的阻塞项不是 UI，也不是模型接入，而是：

1. 数据仓目录协议还没有实现
2. 对象读写层还没有实现
3. AI 执行引擎还没有实现

这三件事不解决，后面做什么都容易返工。

## 12. 第一阶段 MVP 的最小闭环

如果要压缩成最小可用产品，建议先完成这条链：

1. 创建数据仓
2. 采集文本
3. 采集图片
4. 导入文件
5. 生成 `staging/document`
6. 支持 AI 检索
7. 支持发布到 `final`
8. 支持快照和确认

这条链路通了，产品的主价值就已经成立。

## 13. 当前结论

开发优先级最重要的一句话是：

“先把协议、对象、执行控制做出来，再去做复杂界面和边缘功能。”
